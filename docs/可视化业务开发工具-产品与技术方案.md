## 1. 产品概述

### 1.1 产品定位

基于三方完善的业务流程引擎、规则引擎、事件处理低代码/无代码工作流引擎，构建**可视化业务开发工具**，实现业务流程的可视化配置、自动化执行和统一管理。

### 1.2 核心价值

- **降低技术门槛**：业务人员通过可视化界面配置业务流程，无需编码
- **提升开发效率**：标准化流程配置，减少重复开发工作
- **增强业务灵活性**：快速响应业务变化，支持流程动态调整
- **统一管理规范**：基于总线处理机制，规范化业务开发过程

### 1.3 目标用户

- **业务配置人员**：负责业务流程的配置和管理
- **业务运营人员**：使用配置好的流程进行业务操作
- **系统管理员**：管理引擎配置和系统参数
- **开发人员**：进行系统集成和扩展开发

### 1.4 核心功能模块

1. **流程规则引擎**：可视化流程配置与规则管理
2. **事件引擎**：事件驱动的流程触发与流转
3. **自动化引擎消息中心**：流程化消息通知与模板管理
4. **业务主体管理**：业务实体定义与管理
5. **流程设计器**：拖拽式流程可视化设计
6. **规则配置器**：业务规则可视化配置
7. **事件配置器**：事件触发规则配置
8. **数据同步管理**：跨系统数据同步配置

### 2.3 业务流程设计

#### 2.3.1 流程配置流程

```
1. 创建业务流程
   ↓
2. 定义业务主体
   ↓
3. 设计流程节点
   ↓
4. 配置节点规则
   ↓
5. 配置流转条件
   ↓
6. 配置事件触发
   ↓
7. 配置消息通知
   ↓
8. 配置数据同步
   ↓
9. 测试流程
   ↓
10. 发布流程
```

#### 2.3.2 流程执行流程

```
1. 触发流程启动事件
   ↓
2. 创建流程实例
   ↓
3. 初始化业务数据
   ↓
4. 进入起始节点
   ↓
5. 执行节点规则
   ↓
6. 触发节点事件
   ↓
7. 发送消息通知
   ↓
8. 判断流转条件
   ↓
9. 流转到下一节点
   ↓
10. 重复步骤5-9直到结束
   ↓
11. 完成流程实例
```

### 2.4 核心功能详细设计

#### 2.4.1 流程设计器功能

**节点类型**：
- **开始节点**：流程入口，定义流程启动条件
- **结束节点**：流程出口，定义流程结束条件
- **任务节点**：人工任务，需要用户操作
- **自动节点**：自动执行任务，无需人工干预
- **审批节点**：审批任务，支持多人审批、会签、或签
- **条件节点**：条件判断，根据条件分支流转
- **并行节点**：并行执行多个分支
- **汇聚节点**：汇聚并行分支
- **子流程节点**：调用其他流程
- **消息节点**：发送消息通知
- **数据同步节点**：执行数据同步操作
- **脚本节点**：执行自定义脚本

**节点属性配置**：
- 节点基本信息（名称、描述、图标）
- 节点执行规则
- 节点权限配置
- 节点超时设置
- 节点重试策略
- 节点异常处理

**流转线配置**：
- 流转条件表达式
- 流转优先级
- 流转标签

2. **数据验证规则**

   - 范围验证（数值范围、日期范围等）

3. **业务计算规则**
   - 字段计算
   - 聚合计算
   - 业务公式计算

4. **权限校验规则**
   - 角色权限校验
   - 数据权限校验
   - 操作权限校验

**规则配置方式**：
- 可视化表单配置
- 表达式编辑器
- 规则模板选择
- 自定义规则脚本

#### 2.4.3 事件配置器功能

**事件类型**：
- **流程事件**：流程启动、流程结束、节点进入、节点离开等
- **业务事件**：业务数据创建、更新、删除等

#### 3.1.2 技术栈选型

**前端技术栈**：
- **框架**：Vue 3 (Composition API)
- **UI组件库**：Element Plus
- **流程设计器**：基于 bpmn-js 或自定义实现
- **状态管理**：Pinia
- **路由**：Vue Router
- **HTTP客户端**：Axios
- **构建工具**：Vite
- **代码规范**：ESLint + Prettier

**后端技术栈**：
- **运行环境**：Node.js 18+
- **Web框架**：Express.js / Koa.js
- **ORM框架**：Sequelize (MySQL) / TypeORM
- **流程引擎**：基于 Camunda / Activiti 或自研
- **规则引擎**：Drools / 自研规则引擎
- **事件处理**：EventEmitter / 消息队列 (RabbitMQ / Kafka)
- **任务调度**：node-cron / Bull (基于Redis)
- **缓存**：Redis
- **消息队列**：RabbitMQ / Redis Queue

**数据库选型**：
- **关系型数据库**：MySQL 8.0+ (业务数据、流程实例数据)
- **文档数据库**：MongoDB (流程定义、规则定义)
- **缓存数据库**：Redis (缓存、会话、消息队列)

**基础设施**：
- **容器化**：Docker + Docker Compose
- **CI/CD**：GitLab CI / GitHub Actions
- **监控**：Prometheus + Grafana
- **日志**：ELK Stack (Elasticsearch + Logstash + Kibana)

### 3.2 数据库设计

```sql
CREATE TABLE `process_definition` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `code` VARCHAR(100) UNIQUE NOT NULL COMMENT '流程编码',
  `name` VARCHAR(200) NOT NULL COMMENT '流程名称',
  `version` INT NOT NULL DEFAULT 1 COMMENT '版本号',
```

## 5. 推拿师身份迁移流程实现

### 5.1 业务流程分析

#### 5.1.1 流程路径

```
注册 → 核验身份 → 课程学习 → 认证考试 → 成为认证推拿师
```

#### 5.1.2 流程节点详细设计

**1. 注册节点 (register)**
- **节点类型**：自动节点
- **节点功能**：用户注册，创建推拿师账户
- **执行规则**：
  - 验证手机号格式
  - 验证手机号唯一性
  - 验证密码强度
- **事件触发**：
  - 注册成功：触发 `user.registered` 事件
  - 注册失败：触发 `user.register.failed` 事件
- **消息通知**：
  - 注册成功：发送注册成功短信
  - 注册验证码：发送验证码短信
- **数据同步**：
  - 同步用户信息到CRM系统
  - 同步用户信息到认证系统

**2. 核验身份节点 (verify_identity)**
- **节点类型**：任务节点（人工审核）
- **节点功能**：审核推拿师身份信息
- **执行规则**：
  - 验证身份证信息完整性
  - 验证身份证真实性
  - 验证从业资格证（如有）
- **分配规则**：
  - 分配给身份审核专员角色
  - 支持自动分配和手动分配
- **事件触发**：
  - 核验通过：触发 `identity.verified` 事件
  - 核验失败：触发 `identity.verification.failed` 事件
- **消息通知**：
  - 核验通过：发送核验成功通知
  - 核验失败：发送核验失败通知及原因
- **超时处理**：
  - 超时时间：48小时
  - 超时动作：自动提醒审核人员

**3. 课程学习节点 (course_learning)**
- **节点类型**：任务节点（人工+自动）
- **节点功能**：推拿师完成课程学习
- **执行规则**：
  - 验证课程完成度（>= 80%）
  - 验证课程考试通过
  - 验证学习时长（>= 规定时长）
- **事件触发**：
  - 课程完成：触发 `course.completed` 事件
  - 课程进度更新：触发 `course.progress.updated` 事件
- **消息通知**：
  - 课程开始：发送课程开始通知
  - 课程进度提醒：每完成25%发送进度提醒
  - 课程完成：发送课程完成通知
- **数据同步**：
  - 同步学习进度到学习系统
  - 同步学习记录到认证系统

**4. 认证考试节点 (certification_exam)**
- **节点类型**：任务节点（人工+自动）
- **节点功能**：推拿师参加认证考试
- **执行规则**：
  - 验证考试资格（课程完成）
  - 验证考试分数（>= 60分）
  - 验证考试时间（在规定时间内完成）
- **事件触发**：
  - 考试通过：触发 `exam.passed` 事件
  - 考试失败：触发 `exam.failed` 事件
- **消息通知**：
  - 考试开始：发送考试开始通知
  - 考试通过：发送考试通过通知

**5. 成为认证推拿师节点 (become_certified)**
- **节点类型**：自动节点
- **节点功能**：更新推拿师认证状态，授予认证推拿师身份

  - 更新用户认证状态
  - 授予认证推拿师角色
- **事件触发**：
  - 认证完成：触发 `masseur.certified` 事件
- **消息通知**：
  - 认证成功：发送认证成功通知及证书
  - 欢迎消息：发送欢迎加入认证推拿师团队消息
- **数据同步**：
  - 同步认证状态到所有相关系统
  - 同步推拿师信息到推拿师库
  - 同步推拿师信息到订单系统

### 5.3 规则定义配置

#### 5.3.1 注册节点规则

**validate_phone_format**：
```javascript
function validatePhoneFormat(phone) {
  const phoneRegex = /^1[3-9]\d{9}$/;
  return phoneRegex.test(phone);
}
```

**validate_phone_unique**：
```javascript
async function validatePhoneUnique(phone) {
  const user = await db.user.findOne({ phone });
  return user === null;
}
```

**validate_password_strength**：
```javascript
function validatePasswordStrength(password) {
  // 至少8位，包含字母和数字
  const passwordRegex = /^(?=.*[A-Za-z])(?=.*\d)[A-Za-z\d]{8,}$/;
  return passwordRegex.test(password);
}
```

```javascript
function validateIdCardComplete(idCard) {
```

```javascript
async function validateIdCardReal(idCard) {
```

### 5.5 消息模板与通道

- 注册成功：`register_success`（短信）

- 课程开始/进度/完成：`course_started` / `course_progress_reminder` / `course_completed`（App推送+站内信）
- 考试开始/通过/失败：`exam_started` / `exam_passed` / `exam_failed`（短信+App推送）
- 认证成功与欢迎：`certification_success` / `welcome_certified_masseur`（短信+App推送+邮件）
- 运营催办：流程节点超时触发通用催办模板

通道策略：验证码/关键链路优先短信，通知优先站内信/推送，营销优先站内信→推送→短信；支持频控、静默期、黑白名单。

### 5.6 数据同步配置

- `sync_user_to_crm`：注册成功 → CRM 创建客户/线索。
- `sync_learning_progress`：课程进度/完成 → 学习与认证系统。
- `sync_exam_result`：考试成绩 → 认证系统。
- `sync_certification_status`：认证通过 → 账户、权限、结算系统。
- `sync_masseur_to_library` / `sync_masseur_to_order_system`：认证通过 → 推拿师库与订单系统。

机制：幂等键（userId+processVersion）、重试+死信、补偿流程，支持增量+全量校验。

### 5.7 权限与安全控制（落地）

- 节点权限：assignee 支持 user/role/dept/动态变量；进入节点需 RBAC+ABAC+列级校验。
- 数据权限：查询/更新按业务对象套用 ABAC 规则；跨组织按业务属性范围。
- 审计：流程变更、规则变更、模板变更、实例操作全量留痕；敏感字段脱敏。

### 5.8 监控与运营

- 流程可观测：实例列表、节点 Gantt、关键路径耗时、失败/重试、补偿状态。
- 事件监控：事件堆积、死信、重试次数、消费延迟。
- 消息监控：通道成功率、延迟、失败 TopN、频控命中率。
- 规则监控：命中率、执行耗时、错误率，规则变更审计。

### 5.9 SLA 与验收

- 核心链路成功率 ≥ 99.5%；关键通知 95% ≤ 5s 送达。
- 流程发布与回滚 ≤ 5 分钟，支持灰度。
- 业务侧可在无需研发介入下配置 80%+ 常规流程/规则/消息。

## 7. 风险与对策

- **引擎选型与兼容性**：先以轻量自研+开源组件 PoC，接口抽象可替换，限制深度魔改。
- **性能与可用性**：MQ 主题分片、按业务 key 顺序消费；缓存+幂等；背压与限流；多副本部署。
- **规则复杂度**：决策表优先、脚本兜底，增加规则测试与沙箱，版本管理与回滚。
- **数据一致性**：事件驱动+幂等键+重试+死信+补偿流程；必要处引入对账/校验任务。
- **权限错误风险**：默认最小权限，发布前权限校验清单，审计日志与回溯工具。
- **消息触达风险**：多通道路由与降级，频控/静默保护，送达率监控与自动重试。
